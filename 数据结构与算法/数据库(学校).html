# 数据库

数据库(Database,简称DB)是==长期== 储存在计算机内、有==组织==的、可==共享==的==大量==数据集合

# 数据库的特征

  • 数据按一定的数据模型组织、描述和储存
  • 可为各种用户共享
  • 冗余度较小
  • 数据独立性高
  • 易扩展

# 数据库管理系统

  • 数据库管理系统(Database Management System)是一种操纵和管理数据库的大型软件，用于建立、使用和维护数据库

  • 它对数据库进行统一的管理和控制，以保证数据库的安全性和完整性。

  • 数据库定义功能

    • 提供数据定义语言(DDL)
    • 定义数据库重的数据对象

  • 数据操作功能

    • 提供数据操纵语言(DML)
    • 操纵数据实现对数据库的基本操作（查询、插入、删除和修改）

  • 数据库的运行管理

    • 保证数据的安全性、完整性、多用户对数据的并发使用
    • 发生故障后的系统维护

  • 数据库的建立和维护功能

    • 数据库的批量装载
    • 数据库转储
    • 介质故障恢复
    • 数据库的重组织
    • 性能监视等

# 数据库系统

# DBS

数据库系统（Database System，简称DBS）由一个互相关联的数据的集合和一组用以访问这些数据的程序组成。

# DBS的目标

提供一种方便、高效地存取数据库信息的途径

# 数据管理

对数据进行分类、组织、编码、存储、检索和维护，是数据处理的中心问题

# 数据库系统的应用

# 数据库系统的应用

  • 文件处理系统的弊端：
  • 数据冗余和不一致
    • 不同程序员创建，不同文件结构，不同程序设计语言，内容不一致，多处存储地址
  • 数据访问困难
    • 面对不同需求撰写不同程序项目
  • 数据孤立
    • 数据分散且具有不同格式
  • 完整性问题
    • 数据库中存储的数据必须满足某些特定一致性约束
    • 约束不同文件中多个数据项或添加新约束很困难
  • 原子性问题
    • 计算机发生故障时数据会恢复到故障发生前的一致状态
    • 例如: 账户间转账，借贷两种状态要么都发生，要么都不发生
  • 并发访问异常
    • 为提高系统总体性能及加快响应速度，系统允许多个用户同时更新数据
    • 多个数据访问程序间无协调，会导致数据访问异常
      • 例如:
        • 多个职员同时对银行账户进行取款操作
        • 多个学生同时选课
    • 安全性问题
      • 数据访问者具有不同权限

# 实例与模式

  • 模式 schema– 数据库的总体设计称作数据库模式
    • 例如: 数据库中存储的消费者与购买商品的信息，消费者及购买商品间的关系
    • 对应于程序设计语言中的变量声明（以及与之关联的类型的定义）
    • 物理模式: 在物理层描述数据库的设计
    • 逻辑模式: 在逻辑层描述数据库的设计
  • 实例 instance– 特定时刻存储在数据库中的信息的集合称作数据库的一个实例
    • 对应于程序设计语言中某一时刻变量的值
  • 物理数据独立性 Physical Data Independence – 应用程序不依赖于物理模式
    • 程序员使用逻辑模式来构造数据库应用程序
    • 物理模式隐藏在逻辑模式下，通常可以在应用程序不受影响的情况下被更改

数据模型

  • 数据模型（data model）是数据结构的基础

  • 数据模型是一个描述数据、数据联系、数据语义及一致性约束的概念工具的集合

  • 要求

    • 比较真实的模拟现实世界
    • 容易为人所理解
    • 便于在计算机上实现

  • 数据模型的类型

    • 关系模型：用表的集合表示数据和数据之间的关系
    • 实体-联系模型：现实世界是由实体和实体之间的联系构成的，用于数据库设计
    • 基于对象数据模型：实体-联系模型增加了封装、方法和对象标识的扩展
      • 面向对象数据模型
      • 对象-关系数据模型

  • 半结构化数据模型：半结构化数据模型允许相同类型的数据项含有不同的属性集

    的数据说明--xml

  • 其他模型

    • 网状数据模型
    • 层次数据模型

# 数据库语言

  • 数据库系统提供数据定义语言（data-definition language）来定义数据库模式，以及数据操纵语言（data-manipulation language）来表达数据库的查询和更新

  • 数据操纵语言：DML

    • 对数据库进行检索、插入、修改、删除
    • 数据存取功能
    • DML类型
      • 过程化的DML（procedural DML）
        • 指定需要什么数据以及如何获取这些数据
      • 声明式DML（declarative DML，非过程化的DML）
      • 指定需要什么数据

  • 实例----SQL

  • 数据库模式是通过一系列定义来说明的，这些定义由数据库定义语言来表达

    Example:	create table instructor (
    ID                char(5),
    name           varchar(20),
    dept_name  varchar(20),
    salary           numeric(8,2))

  • 存储在数据库中的数值必须满足某些一致性约束（consistency constraint）

    • 域约束
    • 参照完整性
    • 断言
    • 授权

  • DDL以一些指令作为输入，生成的输出存储在数据字典（data dictionary）中

  • 数据字典包含了元数据（metadata），元数据是关于数据的数据

# 设计过程

  • 需求分析
  • 选择数据模型
  • 概念设计阶段
  • 逻辑设计 阶段
  • 物理设计阶段
  • 实体—联系（E-R）模型使用一组称作实体的基本对象，以及这些对象间的联系
    • 实体：现实世界中可区别于其他对象的一件“事情”或一个“物体”
      • 数据库中实体通过属性集合来描述
    • 联系: 几个实体间的关联
    • 同一类型的所有实体的集合称为实体集，同一类型的所有联系的集合称为联系集
  • 规范化
    • 规范化的目标：生成一个关系模式集合，存储信息时没有不必要的冗余，同时又能很轻易地检索数据
    • 设计一种符合适当范式（normal form）的模式，最常用的方法是使用函数依赖（functional dependency）

# 数据存储和查询

  • 存储管理器：数据库系统中负责在数据库中存储低层数据与应用程序以及向系统提交的查询之间提供接口的部件

    • 存储管理部件：权限及完整性管理器、事物管理器、文件管理器、缓冲区管理器

# 事务管理

  • 事物：数据库应用中完成单一逻辑功能的操作集合
  • 事务特性(ACID)
    • 原子性(Atomicity)
      • 事务中包含的所有操作要么全做，要么全不做
      • 原子性由恢复机制实现
    • 一致性(Consistency)
      • 事务的隔离执行必须保证数据库的一致性
      • 事务开始前，数据库处于一致性的状态；事务结束后，数据库必须仍处于一致性状态
      • 数据库的一致性状态由用户来负责，由并发控制机制实现
      • 如银行转帐，转帐前后两个帐户金额之和应保持不变
      • 事务运行过程中允许暂时的不一致
  • 事务特性(ACID)
    • 隔离性(Isolation)
      • 系统必须保证事务不受其它并发执行事务的影响
      • 对任何一对事务T1，T2，在T1看来，T2要么在T1开始之前已经结束，要么在T1完成之后再开始执行
      • 隔离性通过并发控制机制实现
    • 持久性(Durability)
      • 一个事务一旦提交之后，它对数据库的影响必须是永久的
      • 系统发生故障不能改变事务的持久性
      • 持久性通过恢复机制实现

# 数据库体系结构

  • 并行数据库系统
    • 并行系统通过并行地使用多个处理器和磁盘来提高处理速度和I/O速度
    • 粗粒度并行机：少量能力大的处理器组成
    • 大规模并行机、细粒度并行机:大量更小的处理器
  • 系统性能度量
    • 吞吐量：在给定时间段内所能完成任务数量
    • 响应时间：单个任务从提交到完成所需的时间
  • 分布式数据库系统
    • 数据共享
    • 自治性
    • 可用性

# 数据库用户

  • 用户

    • 无经验用户（naïve user）
      • 通过应用系统的用户接口（菜单等）使用数据库
    • 应用程序员(application programmer)
      • 编写应用程序的计算机专业人员
    • 富有经验的用户(sophisticated user )
      • 不通过编写程序同系统交互，通过数据库查询语言表达需求
    • 专业用户（specialized user）
      • 编写专门的、不适合于传统数据处理模式的数据库应用的富有经验的用户，如计算机辅助设计系统、知识库和专家系统等

  • 数据库管理员 DBA

    • DBA:对体统进行集中控制的人

    • DBA的重要性

      • 重要资源
        • 维护整个组织的信息资源
      • 共享资源
        • 多用户共享，需要统一管理、协调、监控

    • DBA职责

    • 建库方面

      • 确定模式、外模式、存储结构、存取策略
      • 负责数据的整理和装入

    • 用库方面

      • 定义完整性约束条件
      • 规定数据的保密级别、用户权限
      • 监督和控制数据库的运行情况
      • 制定后援和恢复策略，负责故障恢复

    • 改进方面

      • 监督分析系统的性能（空间利用率，处理效率）
      • 数据库重组织，物理上重组织，以提高性能
      • 数据库重构造，设计上较大改动，模式和内模式修改

  • 数据库的总体逻辑结构（模式）可以用实体—联系图（ entity-relationship diagram， E-R图）进行图形化表示
